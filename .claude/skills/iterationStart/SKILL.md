# iterationStart

Аргумент — номер итерации N (из `$ARGUMENTS`). Это **полный драйвер итерации**: спека → реализация → ревью → полировка (`/simplify`). Код добивается качества **здесь** — `/iterationClose` потом только церемония без правок кода (доки, ROADMAP, commit-месседж).

## Фазы (по порядку)

### Фаза 0 — Контекст
1. Прочитай строку #N в `ROADMAP.md`: **колонку «🧵 красная нить»** — какой резюме-ключевик итерация обязана сделать демонстрируемым, existence-gate и колонку «Каркас откуда». Новых инструментов в проекте нет по уставу — растёт **техника**; сверься с «Заметками» ROADMAP: у итерации может быть заготовленное микро-решение (№2 — движок path-ассертов; №4 — Phoenix vs χ²/PSI).
2. Прочитай `CLAUDE.md` (north-star раздел «Красная нить» + главные правила: existence-gate правило 1, ни одного нового инструмента правило 2, домен-фикстура и `PathTrace` правило 3, cost-дисциплина правило 4, дисциплина LiteLLM правило 5, швы правило 6, наглядность финала правило 8) и `docs/tech-decisions.md` (сквозные конвенции; появляется на iter 0 — до того конвенции живут в CLAUDE.md).
3. **Если плотно** (особенно тяжёлые итерации — №4 Phoenix: дрейф распределения веток поверх двух пачек трафика; №5 Prefect: замыкание trajectory-eval петли с re-eval challenger-политики + swap alias + hot-reload) — предложи расщепить и **остановись**, прежде чем писать спеку.

### Фаза 1 — Спека
4. Создай `specs/NN/spec.md` (NN двузначный) по шаблону ниже. **Тонко: ≤1 страницы.** Спека обязана **процитировать пункт красной нити** из ROADMAP как цель итерации. Покажи её словами.

### Фаза 2 — Реализация
5. Реализуй по спеке до выполнения **done-gate** (по факту существования). Соблюдай швы (`CLAUDE.md` правило 6): слои, env только через `Settings`, MLflow-хендл на boundary, собственные LLM-вызовы только через `route()`, **источник истины для golden/CI-ассертов — `PathTrace` из domain, не OTel-спаны**; тесты и path-gate с `route()` — в `replay`. Соблюдай дисциплину LiteLLM (правило 5): SDK-only, без callbacks, телеметрия off, версия запиннена. Не отрезай финальную витрину (правило 8): выбирай реализацию, чей результат потом можно показать (пример: Langfuse — через LangGraph-интеграцию, иначе Agent Graph недоступен).
6. **Не запускай live LLM без явной просьбы** (правило 4): live-прогон заявки через граф, перезапись кассет (`LLM_MODE=record|live`), smoke на живом стеке. Нет кассеты в `replay` — остановись и сообщи, не переключайся в `record`. Оценку стоимости называй (весь проект ≈ $1–3).

### Фаза 3 — Ревью и полировка (финальные шаги реализации)
Гоняется, когда done-gate функционально достигнут и код субъективно готов (не по полу-написанному). Цель — **качество кода, НЕ артефакты**: находки живут только в контексте, применяются, испаряются. Никаких `docs/reviews/`.

7. **Два ревьюера параллельно** (в одном сообщении, два вызова Agent — свежий контекст):
   - `general-reviewer` — логика/edge-cases/простота/scope-creep;
   - `constitution-reviewer` — швы `CLAUDE.md` + консистентность с аналогами.
   Каждому передай путь `specs/NN/spec.md`. Дифф они собирают сами с рабочего дерева — НЕ передавай его.
8. **Дедуп.** Одна проблема в одном месте под разными углами → склей, severity = максимум, помечай источник.
9. **Аудитор.** Вызови `review-auditor`, вставив склеенный список находок в промпт. Он выносит KEEP / DOWNGRADE→X / DROP / REWORD — страж против кривого авто-фикса по ложному срабатыванию.

   **Результат аудитора забирай pull, не push.** Push-нотификация о завершении агента иногда не доходит (наблюдалось: агент отработал за ~20с и закончил `end_turn`, но `<task-notification>` не прилетела — выглядит как «зависание», хотя вердикт готов). Поэтому: если нотификация не пришла в разумный срок (~60–90с), **не жди дальше** — вытащи финальный ответ прямо из транскрипта агента по пути `output_file`, который вернул вызов Agent. **Только хирургически через `jq`** (не `Read`/`cat`/`tail` целиком — это переполнит контекст):
   `jq -r 'select(.message.content) | .message.content[]? | select(.type=="text") | .text' <output_file> | tail -40`
   Если у последней записи `stop_reason=end_turn` — агент завершился штатно, бери этот текст как вердикт. Если транскрипт реально оборвался на pending tool_use (а не на тексте) — вот это настоящий зависший вызов: останови агент (`TaskStop`) и применяй **консервативный фолбэк** (см. шаг 10).
10. **Применить вердикты:** DROP → выбросить; DOWNGRADE → сменить severity; REWORD → поправить якорь. Затем **CRITICAL + BUG → чини**; **NIT** → только если правка очевидно безопасна. После фиксов прогони **точечные** тесты (в `replay`), чтобы фикс не сломал happy-path.
   - **Консервативный фолбэк (аудитор реально недоступен).** Если вердикт не достать ни push, ни pull: выступи аудитором сам по его же критериям, но строго в защиту кода — для каждой находки по умолчанию KEEP только то, что чинишь **очевидно безопасно** (NIT-косметика), а на спорное/CRITICAL без явного подтверждения **не накладывай авто-фикс**. Лучше оставить находку на ревью пользователю, чем испортить корректный код. Отметь в хэндоффе, что аудитор был недоступен.
11. Существенные фиксы — можно прогнать ревьюеров повторно. Цель фазы: CRITICAL/BUG = 0.
12. **Причесать код.** Когда ревью чисто, вызови встроенный навык `/simplify` по текущему диффу — косметические правки качества (reuse/simplification/efficiency/altitude) поверх результата ревью, включая код самих фиксов. Это не охота на баги — корректность и швы уже отловлены ревьюерами. После — прогони **точечные** тесты (в `replay`), чтобы полировка не сломала happy-path.

### Фаза 4 — Хэндофф
13. Короткая сводка: done-gate выполнен; **какой пункт красной нити стал демонстрируемым**; ревью — 2-4 строки (что починил по severity, что аудитор отбросил); что причесал `/simplify`; осознанные срезы, если решили что-то НЕ чинить (пойдут в `learnings.md` при close).
    - **Итерация ещё НЕ закрыта.** Реализация и ревью — готовы, но ROADMAP **остаётся ⬜**, доки не сгенерированы, изменения не застейджены, пока не прогнан `/iterationClose N`. Явно скажи это пользователю и предложи следующий шаг — `/iterationClose N`.

## Шаблон `specs/NN/spec.md`

```markdown
# Итерация NN — <короткое имя>

> 🎯 Новая техника на старом стеке, минимальными затратами. Existence-gate, не accuracy-gate.

## Цель
<1-2 предложения: какую технику вводим и зачем именно она>

## 🧵 Красная нить (резюме)
<дословно строка из ROADMAP: какой trajectory-eval-ключевик эта итерация делает демонстрируемым>

## Новая техника (и минимальный объём)
- **<техника>** — <что именно делаем, насколько минимально; какой каркас переиспользуем>

## Done-gate (по факту существования)
<конкретная проверяемая вещь: golden-сет лежит как Evaluation Dataset с expectations / path-gate валит CI при регрессии маршрутизации / SLO-лог называет просевшую ноду + Agent Graph виден / Phoenix показывает дрейф распределения веток / swap policy-alias → hot-reload>
+ ревью-пайплайн чист (CRITICAL/BUG = 0).
<если итерация сеет/мутирует состояние стора (заливка golden-сета, регистрация промптов/policy-версии, промоушен/swap alias) — команда обязана быть **идемпотентной**: повторный прогон = no-op или заявленное стабильное состояние, без распухания версий и дрейфа alias. Иначе done-gate держится только на свежем сторе и живой смок закрытия его не воспроизведёт.>

## Шаги
1. ...
2. ...
(3-5 пунктов; последний — ревью-пайплайн)

## Вне scope
<что НЕ трогаем в этой итерации, чтобы не уплотнять>
```

## Правила
- Спека тонкая (≤1 страницы). Не раздувай — детали оркестрации ревью живут здесь, в скилле, не в спеке.
- Не предлагай live LLM-шаги без явной пометки про стоимость.
- Микро-решения по ходу (напр. маппинг тира на конкретный снапшот модели; состав «джокеров» golden-сета — помни квоту ≥80% singleton, правило 3; движок path-ассертов в №2; Phoenix vs χ²/PSI в №4) — спрашивай, если развилка существенна; заготовленные варианты смотри в ROADMAP → Заметки.
