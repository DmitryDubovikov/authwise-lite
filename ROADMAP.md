# authwise-lite — ROADMAP

> 🎯 **Цель проекта:** минимальными затратами — максимальная демонстрация trajectory-eval.
> Один-единственный новый инструмент на весь проект (Prometheus/Grafana, iter 4 — Заметки);
> в остальном одна новая **техника** на итерацию, проверка по факту существования, не по
> точности.

Источник истины по **порядку и составу**. Архитектурные правила — в `CLAUDE.md`. Тонкая спека
каждой итерации создаётся `/iterationStart N` в `specs/NN/`.

Статусы: ⬜ todo · 🟦 in progress · ✅ done.

> Бюджет: **6 содержательных итераций + каркас (iter 0) + 1 опц. хвост = 7–8**, потолок ниже,
> чем у triagewise-lite (9). Это осознанно: почти весь тулинг здесь **переиспользуется** из
> policywise-lite (LangGraph, LiteLLM, кассеты) и triagewise-lite (MLflow, promptfoo/DeepEval,
> OTel, Langfuse, Phoenix) — новых инструментов не вводим, **с одним осознанным исключением:
> Prometheus/Grafana (iter 4, Заметки)**; по правилу «новый инструмент → каркас на его
> освоение» исключение получает собственную итерацию. Растёт в остальном только **техника**:
> объект измерения — путь агента по графу, а не финальный ответ и не отдельный prompt-вызов.
> Рост против первоначальной оценки (5–6): iter 4 добавлен под новый инструмент, iter 6
> (registry-механика routing-policy) выведен из опционального хвоста — он закрывает строку
> резюме и опциональным быть не может.
>
> Закон ROADMAP (наследуется от triagewise-lite): **колонка «🧵 красная нить» обязательна** для
> каждой итерации — она называет резюме-практику (trajectory-eval), которую итерация делает
> демонстрируемой. Итерация без записи в этой колонке не планируется.

## Домен / фикстура (data-prep)

Вымышленный payer **Northfield Health**, поток **Prior Authorization (PA)**-заявок. Артефакты:
- **Поток заявок** — синтетические PA-реквесты (English), ~2 пачки: базовая + «пострелизная»
  (вводит сдвиг в распределении причин заявок — демо path-drift, аналог новой категории в
  triagewise). Владельцы по итерациям: iter 0 — только smoke-фикстура 3–5 заявок; базовая
  пачка ~30 создаётся и размечается в iter 1; «пострелизная» ~30 — в iter 5.
- **Trajectory golden-сет (~30 заявок, размеченный)** — эталон не финального решения, а **пути**:
  ожидаемая ветка (`approve` / `request-info` с числом циклов / `escalate`) для каждой заявки, с
  допустимыми альтернативными путями там, где заявка объективно неоднозначна. Хранится как
  **MLflow Evaluation Dataset** — штатная сущность MLflow ≥3.4 для эталонов с ожидаемыми
  результатами (expectations); наш sqlite-бэкенд из slim-Compose её поддерживает.
- **Граф (заморожен после iter 0):** `classify → policy-check → decide{approve | request-info
  (retry-loop, до N попыток) | escalate}`. Новых веток не добавляем — усложняем измерение, не
  граф.

## Сквозные контракты (решены 2026-07-06, до iter 0)

Контракты, которые потребляются **несколькими** итерациями, — фиксируются здесь, чтобы каждый
`/iterationStart` не решал их заново и по-своему. Внутренности одиночных итераций (тексты
промптов, маппинг полей Evaluation Dataset, wiring Langfuse/Phoenix, панели Grafana) здесь
НЕ фиксируются — это скоуп тонких спек.

1. **Схема заявки-фикстуры** (iter 0/1/2/5): один формат для всех пачек — JSONL,
   `{"id": "PA-<pack>-<NNN>", "text": ..., "meta": {...}}`, pack ∈ {`smoke`, `base`, `post`}.
2. **Семантика `PathTrace`** (все итерации): **три терминала** — `branch` ∈ {`approve`,
   `escalate`, `request-info`}; терминальный `request-info` = «недостающие документы не получены
   после N до-запросов»; `retry_cycles` — число прокруток retry-loop до терминала; исчерпание
   *бюджета* (iter 4) → `escalate`. Ассерты — только по `(branch, retry_cycles)`; `nodes` —
   информационное поле (витрина, Langfuse), в golden/CI не ассертится. **Потолок retry N=2** —
   фиксируется до разметки golden (iter 1), от N зависят ожидаемые `retry_cycles`.
3. **`RunRecord` — единый артефакт батч-прогона** (iter 2/4/5/7): один раннер в workflow-слое
   гонит пачку через граф и пишет JSONL `{request_id, path_trace, per-node usage/latency}`;
   CI-таблица, Prometheus-метрики, Phoenix-датасеты и Prefect re-eval **читают его**, а не
   гоняют граф каждый по-своему.
4. **Раскладка кассет** (iter 0/2/5): каталог на набор — `cassettes/<set>/`
   (`base`, `base-broken-policy`, `post`), набор выбирается env (`AW_CASSETTE_SET`); ключ внутри
   набора — `(request_id, node, attempt)`, **не хэш содержимого запроса** — это структурное
   решение ловушки демо-регрессии iter 2 (Заметки).
5. **Цены и стоимость** (iter 3/4): цены токенов — в `llm-tiers.yaml` рядом с моделями
   (актуализировать при смене снапшота); cost ноды = `usage` × цена — **одна** функция в domain,
   Prometheus и SLO-расчёты используют её; **бюджет рана — в USD** (`AW_RUN_BUDGET_USD`),
   резюме-строка «cost-bounded retry loop» остаётся честной.
6. **Детерминизм**: `temperature=0` во всех нодах; record-прогоны только при temperature=0 —
   иначе `retry_cycles` в golden невоспроизводимы.
7. **Env-конвенция**: все переменные — через `Settings` с префиксом `AW_`
   (`AW_LLM_MODE`, `AW_CASSETTE_SET`, `AW_RUN_BUDGET_USD`, `AW_RETRY_LIMIT`, `AW_TIER_*`).

## Контроль красной нити (анти-дрейф)

- На `/iterationStart N`: спека **цитирует** строку из таблицы «🧵 красная нить» как цель итерации.
- На `/iterationClose N`: в `docs/` фиксируем, какой resume-ключевик стал демонстрируемым.
- Финальный showcase-README: каждая практика north-star → строка таблицы со ссылкой на
  доказательство, как в policywise/triagewise. Стек-строка резюме собирается **только** из
  реально продемонстрированного — и явно помечена как «то же оборудование, применённое к
  траектории», чтобы не читалось как дубль triagewise.

## Отношение к policywise-lite и triagewise-lite

Явно не самостоятельный домен «с нуля»: граф — переиспользование паттерна policywise, весь
eval-ops тулинг — переиспользование стека triagewise. Единственное новое — объект измерения
(путь вместо ответа). Это осознанное решение: три существующих проекта уже показывают
architecture (policywise), multi-agent orchestration (dossier), single-call LLMOps (triagewise);
четвёртый закрывает ровно то пересечение, которого не хватало — trajectory-level eval поверх
многошагового агента, — не добавляя новой оси инструментов.

## Бюджет / стоимость

Cloud-only через тиры, как в triagewise. Кассеты `replay`=$0 (дефолт); `live`/`record` гейтится
(спросить + оценка). Прогноз всего проекта: **~$2–4** — три плановых record-прогона (базовая
пачка в iter 1, «сломанный» policy-check в iter 2, «пострелизная» пачка в iter 5; каждый
копеечный на `cheap`-тире) плюс демо-прогоны. Расход на демо-регрессию iter 2 одобрен
пользователем (2026-07-06).

## Заметки

- **№2 — движок path-ассертов: рекомендация — pytest (пересмотрено 2026-07-03 после сверки с
  практикой).** Отрасль тестирует траектории LangGraph через pytest + trajectory-evaluator
  (langchain-ai/**agentevals**: graph trajectory match по посещённым нодам, режимы
  strict/unordered/subset/superset) — не через promptfoo. Наш ассерт примитивен (membership
  `PathTrace` в `allowed_paths` — три строки), поэтому: **pytest + собственный ассерт** —
  рекомендация (ноль новых зависимостей); agentevals — отраслевая референс-точка, не тащим
  (устав «ни одного нового инструмента»); promptfoo — наиболее искусственный вариант (путь
  пришлось бы протаскивать через кастомный provider), понижен. Финализация на
  `/iterationStart 2`; красная нить (CI красный при регрессии маршрутизации) от движка не зависит.
- **№2 — ловушка replay при демо-регрессии (решено 2026-07-06).** Кассеты ключуются по
  содержимому запроса → «сломанный» промпт policy-check в replay дал бы **cassette-miss, а не
  другой маршрут**: CI покраснел бы по неправильной причине, и таблица «ожидаемый vs фактический
  путь» не собралась бы. Решение: для сломанной версии записываются **свои** record-кассеты
  (гейтированный расход одобрен), демо-ветка коммитит их вместе с изменённым промптом; CI
  остаётся $0-replay, регрессия — честно маршрутная. Следствие для iter 0: ключевание/раскладка
  кассет должны допускать несколько наборов рядом (например, каталог кассет на вариант промпта).
- **№4 — Prometheus/Grafana: единственное исключение из «ни одного нового инструмента»
  (решение 2026-07-06).** Причины: пары нет ни в одном сиблинге при максимальной частотности в
  вакансиях; «SLO» без alert rule — просто лог. Исключение одно и получает собственную
  итерацию — правило «новый инструмент → каркас на освоение» работает в обе стороны. Оговорки:
  в replay латентность ~0 → демо алерта через ужатый порог; cost — из `usage`, сохранённого в
  кассетах (требование к формату — iter 0); способ экспорта метрик (prometheus-client vs
  OTel-экспортер) — деталь `/iterationStart 4`; частичное перекрытие с Langfuse осознанное:
  Langfuse = трейсинг/атрибуция (iter 3), Prom/Grafana = SLO/алертинг (iter 4).
- **№4 — runtime budget controls меняют контракт `decide` (зафиксировано в CLAUDE.md,
  правило 3).** Было: детерминированная функция только над структурированным выходом
  policy-check; стало: **плюс остаток бюджета рана** — retry только при положительном остатке,
  исчерпание → `escalate`. Новых веток и полей `PathTrace` не появляется (исчерпание видно
  через ветку + `retry_cycles`); дефолтный бюджет калибруется так, чтобы golden-пути не
  менялись; демо — ужатый бюджет через env.
- **DVC осознанно не тянем.** В triagewise golden-сет жил в DVC, а промпт — в MLflow. Здесь
  golden-путь **и есть** версионируемый артефакт реестра — это красная нить iter 1; второй стор
  версионирования дублировал бы демонстрацию, ничего не добавляя к north-star.
- **№5 — дрейф здесь категориальный, не embedding; Phoenix проверен (2026-07-03).** Объект —
  распределение выбранных веток (доли `approve`/`request-info`/`escalate`), а не дрейф
  эмбеддингов/качества, как в triagewise. По докам: **Phoenix Inferences** (`px.Schema`,
  primary vs reference) поддерживает структурные/категориальные колонки (features/predictions)
  **без эмбеддингов** и даёт drift-сравнение двух inference-сетов. План: ветка из `PathTrace` —
  категориальная колонка; базовая пачка = reference, «пострелизная» = primary. Оговорки:
  Inferences — легаси-уголок Phoenix (актуальный фокус — tracing/evals; страницы доков
  переезжают) → **версию пиннить**; это batch-анализ двух наборов, а не стриминговый мониторинг —
  для existence-gate достаточно, в доках не называть «real-time». Fallback (χ²/PSI своими руками
  + визуализация) остаётся на случай, если в OSS-UI не окажется per-feature drift-карточки.
  Финальная механика — `/iterationStart 5`.
- **Развилки, решённые до старта (2026-07-03) — полные формулировки в CLAUDE.md → «Стек:
  развилки уже решены»:** провайдер — OpenAI (тиры/пиннинг из triagewise); `PathTrace` —
  first-class domain-объект (OTel — не источник истины ассертов); golden-семантика — membership
  + квота ≥80% singleton (отраслевая практика trajectory-eval: exact-match хрупок); golden-сет
  хранится как **MLflow Evaluation Dataset** (штатная сущность ≥3.4), не как трюк поверх реестра
  промптов.
- **«Routing-policy» = решение «Г», двухуровневое (штатный механизм MLflow 3):** промпты
  `classify`/`policy-check` версионируются в Prompt Registry по отдельности; поверх — версия
  приложения (LoggedModel), пинящая конкретные версии промптов; alias `champion`/`challenger` —
  на ней, swap атомарный. До iter 6 промпты живут в коде (iter 2 ломает промпт policy-check
  прямо в рабочем дереве — реестр гейту не нужен); регистрация + пиннинг + alias-загрузка +
  ручной swap — скоуп iter 6 (**не** опц.: закрывает строку резюме про routing-policy);
  автопромоушен по расписанию — iter 7 (опц.). Fallback при разбухании iter 6: alias на одном
  промпте policy-check с пометкой `# aw-lite:`.
