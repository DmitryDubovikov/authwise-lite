# Демо итерации 02 — CI path-assertion gate / trajectory regression testing

Прогон доказывает вторую north-star-практику проекта — **trajectory regression testing**: у команды
появился автоматический гейт, который краснеет, когда агент начинает ходить по другому маршруту.
Ценность за пределами демо: это фундамент против тихих регрессий маршрутизации — правка промпта,
которая уводит заявку не туда, ловится в CI до мёрджа, а не в проде. Демо показывает три вещи по
очереди: (1) на текущем `main` гейт зелёный — пути пачки точно совпадают с эталоном; (2) гейт умеет
краснеть **именно из-за смены маршрута**, а не из-за отсутствия кассеты (нарочно «сломанный» набор);
(3) артефакт гейта — таблица «ожидаемый vs фактический путь» — читаемый материал для витрины.

Все команды выполняются из корня репо: `/Users/dd/projects/pet/authwise-lite`. Вся итерация
**бесплатная** — LLM не зовётся, всё в replay ($0). Live-шагов нет; единственный ручной шаг с
внешним эффектом (branch protection) вынесен в конец и требует подтверждения.

## 1. Окружение

Зачем: остальные шаги гоняются через `uv run` из локального venv.

```bash
uv sync --extra dev
```

**Ожидаемо:** завершается без ошибок (venv `.venv/` создан/обновлён из `uv.lock`).

## 2. Статический гейт (то, что гоняет CI) — включая path-assertion gate

Зачем: `make check` — это ровно то, что гоняет CI-джоб; в него теперь входит `tests/test_path_gate.py`
— path-assertion gate на всех 30 golden-заявках. Всё офлайн, replay, без docker и сети.

```bash
make check
```

**Ожидаемо:** все четыре шага зелёные (ruff check, ruff format, mypy, pytest), последняя строка
pytest — `40 passed`. Среди них 4 теста `tests/test_path_gate.py` (base проходит, сломанный набор
краснеет из-за маршрута, контроли зелёные, JSONL round-trip) и 2 smoke-теста, переведённые на общий
раннер.

## 3. Гейт на текущем `main` — зелёный, с таблицей путей (это и есть артефакт)

Зачем: показать **сам артефакт гейта** — таблицу «ожидаемый vs фактический путь» по всей базовой
пачке. Это то, что печатается в лог CI и идёт на витрину; и это подтверждение, что на `main` маршруты
совпадают с эталоном из iter 1.

```bash
make path-gate
```

**Ожидаемо:** заголовок `набор кассет: base`, затем 30 строк `PA-base-0NN  ✓ ok  ожид: … факт: …`
(включая джокеров вроде `PA-base-027  ожид: approve ↻0 | approve ↻1  факт: approve ↻0`), финальная
строка `PASS`, exit-код `0`. Ни одной строки `✗ REGRESSION`.

## 4. Демо-регрессия — гейт КРАСНЕЕТ из-за смены маршрута (ключевой шаг)

Зачем: доказать, что гейт ловит именно **регрессию маршрутизации**, а не что-нибудь ещё. На нарочно
«сломанном» наборе кассет (policy-check штампует всем `sufficient`) три заявки уходят с golden-ветки.
Критично, что краснота — из-за пути, а не из-за отсутствия кассеты: то, что гейт **доходит до
таблицы**, уже означает, что replay нашёл кассеты (miss был бы громкой ошибкой раньше).

```bash
make path-gate-broken
```

**Ожидаемо:** заголовок `набор кассет: base-broken-policy`, 5 строк subset:
`PA-base-001 ✓ ok`, `PA-base-003 ✓ ok` (контроли — маршрут не изменился), затем три регрессии —
`PA-base-015 ✗ REGRESSION  ожид: approve ↻1  факт: approve ↻0` (потерян retry-цикл),
`PA-base-019 ✗ REGRESSION  ожид: request-info ↻2  факт: approve ↻0` (смена ветки),
`PA-base-021 ✗ REGRESSION  ожид: escalate ↻0  факт: approve ↻0` (смена ветки). Финал —
`FAIL — 3 регрессий маршрута`; команда завершается ненулевым кодом (`make` рапортует `Error 1`).

## 5. Тот же красный результат как pytest (то, что реально валит CI)

Зачем: `make path-gate` — витрина, а блокирует мёрдж именно pytest внутри `make check`. Показать, что
регрессия выражена как падающий тест, а не только как таблица в stdout.

```bash
uv run pytest tests/test_path_gate.py -v
```

**Ожидаемо:** `4 passed`. Тест `test_broken_set_regresses_on_route_change` **проходит** — потому что
он утверждает, что на сломанном наборе гейт краснеет ровно на `{PA-base-015, PA-base-019, PA-base-021}`.
(Зелёный тест здесь = «гейт корректно обнаружил регрессию»; красным он стал бы, если бы гейт
*перестал* её ловить.)

## 6. Артефакт прогона (JSONL) — контракт №3 для следующих итераций

Зачем: показать файловый артефакт, который `path_gate` пишет попутно, — его будут читать Prometheus
(iter 4), Phoenix (iter 5), Prefect (iter 7), не гоняя граф заново.

```bash
head -2 runs/base.jsonl
```

**Ожидаемо:** две JSONL-строки вида
`{"path_trace": {"branch": "approve", "nodes": ["classify", "policy-check", "decide"], "retry_cycles": 0}, "request_id": "PA-base-001"}`.
Каталог `runs/` gitignored (артефакт прогона, не исходник).

## 7. Идемпотентность authored-кассет (вспомогательная проверка)

Зачем: «сломанный» набор кассет генерится скриптом; убедиться, что повторный прогон не плодит
изменений (детерминированный, стабильный набор в репо).

```bash
make author-broken-cassettes && git status --porcelain cassettes/base-broken-policy/
```

**Ожидаемо:** скрипт печатает `wrote cassettes/base-broken-policy/…` по каждому файлу; `git status`
после — либо пусто (файлы уже закоммичены и не изменились), либо только `?? cassettes/base-broken-policy/`
как untracked-каталог, но **без** модификаций содержимого. Повторный прогон детерминирован.

## 8. ⚠️ Ручной шаг (внешний эффект, НЕ в церемонии закрытия): branch protection

Зачем: чтобы красный гейт **блокировал мёрдж**, GitHub должен требовать статус-контекст `check` —
это имя *job* в `.github/workflows/ci.yml` (`jobs: check:`), а не имя workflow (`ci`). Это меняет
поведение push/merge в `main`, поэтому включается вручную, один раз, с подтверждением — не кодом и
не церемонией закрытия.

```bash
# требует gh с правами admin на репо; выполнить один раз при желании включить блокировку мёрджа
gh api -X PUT repos/DmitryDubovikov/authwise-lite/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  -f 'required_status_checks[strict]=true' \
  -f 'required_status_checks[contexts][]=check' \
  -f 'enforce_admins=true' \
  -f 'required_pull_request_reviews=' \
  -f 'restrictions='
```

**Ожидаемо:** JSON-ответ с настройками protection, где `required_status_checks.contexts` содержит
`check`. После этого PR с красным CI нельзя смёрджить. **В церемонию закрытия этот шаг не входит** —
приведён для полноты north-star (branch protection блокирует мёрдж).
