# Learnings — итерация 01

## Ключевые решения и почему

- **Golden-сет = MLflow Evaluation Dataset, не DVC и не файл в логах.** Развилка была решена ещё
  до старта (ROADMAP → Заметки: «DVC осознанно не тянем»), но итерация её материализовала. Логика:
  в triagewise golden-сет жил в DVC, а промпт — в MLflow; здесь golden-*путь* **и есть** красная
  нить iter 1, и версионировать его в реестре — прямее, чем поднимать второй стор. Ново только
  *что* лежит в реестре (путь), сам MLflow — перенос из triagewise, нового инструмента нет.

- **Membership-ассерт, а не exact-match.** `path_allowed()` засчитывает путь, если пара
  `(branch, retry_cycles)` входит в `allowed_paths`. Exact-match по одной эталонной траектории —
  хрупкий (отраслевая практика trajectory-eval допускает несколько reference-траекторий). Квота
  ≥80% singleton не даёт разметке «размякнуть»: список из двух путей разрешён только там, где
  заявка объективно неоднозначна, и джокер обязан нести `note` (валидатор `_honest_markup` роняет
  джокера без объяснения).

- **`nodes` в ассерт не входят — только `(branch, retry_cycles)`.** Контракт №2 фиксирует `nodes`
  как информационное поле (витрина/Langfuse). Поэтому `AllowedPath` и membership оперируют парой,
  а не тройкой — golden-разметка не завязана на точную последовательность нод, только на терминал
  и число retry-циклов.

- **Идемпотентность — get-or-create + `merge_records`.** В triagewise был реальный баг:
  неидемпотентный `register_prompt` уводил `champion` с v1 на v5. Здесь заранее: датасет ищется по
  имени (`get_dataset`, на `RESOURCE_DOES_NOT_EXIST` — `create_dataset`), а `merge_records` апсертит
  по хэшу inputs. Проверено вживую при закрытии: тройная заливка подряд оставляет ровно 30 записей,
  verify каждый раз `OK`. Плюс офлайн-тест `test_upload_is_idempotent` на временном sqlite —
  двойная заливка не плодит записи.

- **Verify читает стор, не файлы.** `verify_golden()` берёт данные из `golden_store.fetch()`
  (MLflow `to_df`), а не из локального `golden-base.jsonl`, и считает квоту/покрытие по стору
  (правило 9). `expected_ids` для сверки берутся из пачки заявок — так verify ловит и `missing`
  (не долилось), и `extra` (осиротевшие записи после правки inputs).

## Грабли

- **Проще ожидаемого: Evaluation Dataset API минимален.** `mlflow.genai.datasets` даёт ровно то,
  что нужно — `create_dataset` / `get_dataset` / `merge_records` / `to_df`. Отдельного «версионного»
  танца не потребовалось: сущность сама версионируемая, sqlite-бэкенд slim-Compose её держит без
  доп. конфигурации. Пин `mlflow>=3.4,<4` (Evaluation Datasets — сущность 3.x) закрыл вопрос.

- **Сирота после правки inputs — осознанное поведение, не баг.** `merge_records` апсертит по хэшу
  inputs; если поменять `text`/`supplemental` уже залитой заявки, старая запись останется в сторе
  и verify покажет её как `extra` («лишние в сторе»). Чистка — вручную или новым датасетом. Для
  iter 1 этого достаточно: разметка стабильна, а verify честно сигналит рассинхрон вместо тихого
  накопления. Задокументировано в docstring `app/persistence/golden.py`.

## Осознанные срезы и отложенные TODO

- **Граф по пачке в этой итерации не гоняется.** `PathTrace` здесь — только тип для membership;
  прогон 30 заявок через граф в replay и сравнение пройденного пути с эталоном — скоуп iter 2
  (CI path-assertion gate). Это не срез качества, а граница по ROADMAP.

- **Record-кассеты базовой пачки записаны здесь как плановый расход.** `cassettes/base/` (73 файла
  с реальным `usage`, cheap-тир, ≈$0.01) закрывают `# aw-lite: авторские кассеты → реальный record`
  из iter 0. Сам golden-эталон от них не зависит (LLM в iter 1 не зовётся), но раннеру iter 2 они
  нужны для $0-replay-прогона пачки. Records-прогон одобрен и выполнен в рамках реализации итерации.

- **Ручная чистка сирот в сторе не автоматизирована** (см. «Грабли»). Пока разметка стабильна —
  не нужно; если в iter 5 «пострелизная» пачка потребует пересборки датасета, вернуться к вопросу
  «версия датасета vs новый датасет».

- **Existence-gate, не accuracy.** Конкретные ожидаемые пути 30 заявок (какая заявка → approve, а
  какая → escalate) размечены по здравому смыслу PA-домена, не выверялись против реальной политики
  Northfield (её нет — payer вымышлен). Правило 1: качество PA-решений — не ворота; демонстрируемо
  то, что эталон путей существует, версионирован и verify его поднимает из стора.
