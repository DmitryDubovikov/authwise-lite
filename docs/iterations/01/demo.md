# Демо итерации 01 — trajectory golden-сет как артефакт реестра

Прогон доказывает первую north-star-практику проекта — **trajectory-as-artifact**: эталон правильных
маршрутов агента живёт не файлом в логах, а **версионируемой сущностью MLflow** (Evaluation
Dataset), и его содержимое — *сами пути* заявок — можно поднять обратно из стора. Без этого эталона
CI-гейту против регрессий маршрутизации (iter 2) не с чем сравнивать. Вся итерация **бесплатная**:
LLM не зовётся, работаем с фикстурами и реестром; live-шагов нет.

Все команды выполняются из корня репо: `/Users/dd/projects/pet/authwise-lite`.

## 1. Окружение

Зачем: остальные шаги гоняются через `uv run` из локального venv.

```bash
uv sync --extra dev
```

**Ожидаемо:** завершается без ошибок (venv `.venv/` создан/обновлён, зависимости из `uv.lock`,
включая `mlflow>=3.4` — Evaluation Datasets появились в MLflow 3.x).

## 2. Статический гейт + офлайн-гейт разметки

Зачем: это тот же гейт, что гоняет CI; в него входят тесты честности golden-разметки (биекция с
пачкой, квота singleton ≥80%, джокеры ровно поимённые) и идемпотентность заливки на временном
sqlite-сторе — всё офлайн, без docker и сети.

```bash
make check
```

**Ожидаемо:** все четыре шага зелёные (ruff check, ruff format, mypy, pytest), последняя строка
pytest — `36 passed`. Среди них 9 тестов `tests/test_golden.py`.

## 3. Поднять control-plane backend

Зачем: golden-сет заливается в MLflow; шаги 4–6 требуют живого стора.

```bash
make up && sleep 5 && curl -s http://localhost:5051/health && echo " OK"
```

**Ожидаемо:** `OK` (HTTP 200 от MLflow v3.4.0 на порту 5051; 5050 занят MLflow сиблинга triagewise).
Если контейнер уже поднят — `make up` — no-op, health всё равно `OK`.

## 4. Залить golden-сет в реестр (это и есть trajectory-as-artifact)

Зачем: эталон путей превращается из файла в репо в **версионируемую сущность реестра** — ровно тот
шаг, который делает практику демонстрируемой. Иначе эталону путей негде жить как артефакту.

```bash
make golden-upload
```

**Ожидаемо (дословно):**

```
залито 30 записей в Evaluation Dataset 'pa-trajectory-golden-base' (merge — идемпотентно)
```

## 5. Verify В СТОРЕ — и показать сам путь каждой заявки

Зачем: правило 9 — доказываем запросом к самому MLflow, не по логам заливки и не через UI. Шаг
печатает **содержимое артефакта** — пройденные ветки и число retry-циклов, поднятые из стора, — а
не только «датасет существует».

```bash
make golden-verify
```

**Ожидаемо:** 30 строк вида `PA-base-NNN: <ветка> ↻<циклы>`, четыре из них помечены `[джокер]`
(две допустимые траектории), затем сводка. Ключевые строки:

```
PA-base-001: approve ↻0
PA-base-015: approve ↻1
PA-base-019: request-info ↻2
PA-base-021: escalate ↻0
PA-base-027: approve ↻0 | approve ↻1  [джокер]
PA-base-030: approve ↻1 | request-info ↻2  [джокер]

singleton 87% (квота ≥80%)
────────────────────────────────────────────────────────────
✅  VERIFY OK — pa-trajectory-golden-base: 30 записей (ожидалось 30), расхождений с пачкой нет
```

Яркий вердикт последней строкой — сознательно: построчный дамп выше показывает *сам артефакт*
(правило демо), а `✅ VERIFY OK` / `❌ VERIFY FAILED` (ненулевой exit) — вердикт по стору
(правило 9). Вердикт говорит ровно то, что verify доказывает, — **стор совпал с пачкой**
(число записей + ноль расхождений). `singleton %` печатается **справочной** строкой выше
разделителя, а не как вердикт: это метрика строгости разметки, чей настоящий гейт — pytest
(`test_singleton_quota_holds`, офлайн по файлу). На расхождении команда печатает
`❌ VERIFY FAILED` и падает с кодом 1.

## 6. Идемпотентность заливки

Зачем: `golden-upload` — единственный state-мутирующий шаг итерации; повторный прогон обязан быть
no-op (get-or-create датасета + merge_records апсертит по хэшу inputs). Иначе done-gate держался бы
только на свежем сторе, а на второй заливке записи бы распухли.

```bash
make golden-upload && make golden-verify | tail -2
```

**Ожидаемо:** та же строка «залито 30 записей …», а verify по-прежнему `30 записей (ожидалось 30)`
и `✅ VERIFY OK` — число записей не выросло, датасет-дубли не появились.

## 7. Артефакт в репо: разметка джокера с объяснением

Зачем: source-of-truth разметки версионируется в git; джокер обязан нести `note`, объясняющий
неоднозначность (правило 3 — честность разметки). Смотрим запись `PA-base-030` (пример из шага 5).

```bash
jq -c 'select(.request_id=="PA-base-030")' fixtures/golden-base.jsonl
```

**Ожидаемо:** JSON с двумя `allowed_paths` (`approve ↻1` и `request-info ↻2`) и непустым `note` про
CGM при basal-only инсулине — та самая объективная неоднозначность, ради которой заявка помечена
джокером (корм для champion/challenger-гейта, iter 6).
